# ZeroCode 设计文档

**日期**: 2026-03-05

---

## 1. 项目概述

ZeroCode 是一个 Claude Code 任务管理与执行插件，让 Agent 可以持续性地、不间断地执行完项目所有任务。任务执行过程中出现问题时，Agent 会自行分析问题，找到解决方案并执行，直到问题解决为止，然后再执行下一个任务。

## 3. 核心执行流程

```
开始执行
    │
    ├─► 阶段1: 需求分析 ──► 产品经理 (林若曦)
    │     ├─► 领取需求分析任务
    │     ├─► MCP 创建新会话
    │     ├─► 子代理执行任务
    │     └─► 完成 → MCP 创建新会话 → 阶段2
    │
    ├─► 阶段2: 计划制定 ──► 技术经理 (张伟)
    │     ├─► 领取计划制定任务
    │     ├─► MCP 创建新会话
    │     ├─► 子代理执行任务
    │     └─► 完成 → MCP 创建新会话 → 阶段3
    │
    ├─► 阶段3: 任务执行 ──► 程序员 (李明)
    │     ├─► 选择任务 ──► 自动领取机制
    │     │               └─► 递归找到最小可执行任务
    │     │
    │     ├─► MCP 创建新会话
    │     │
    │     ├─► 检查依赖 ──► 递归执行依赖任务（可并行）
    │     │               └─► 依赖任务成功后返回
    │     │
    │     ├─► 检查子任务 ──► 递归执行所有子任务（可并行）
    │     │                   └─► 所有子任务成功后返回
    │     │
    │     ├─► TDD: 生成验收测试用例
    │     │
    │     ├─► 执行任务（调用 Skill）
    │     │
    │     ├─► 状态 → testing ──► 运行测试
    │     │           ├─► 失败 → 根因分析 → 修复 → 重测 (循环)
    │     │           └─► 通过 → 检查完成条件
    │     │
    │     ├─► 检查完成条件
    │     │     ├─► 不满足 → 修复 → 重检 (循环)
    │     │     └─► 满足 → 状态 → completed
    │     │
    │     └─► 继续下一个任务（或并行执行无依赖任务）
    │
    └─► 所有任务完成
```

**执行顺序**: 需求分析 → 计划制定 → 任务执行

## 3.1 任务自动拆解机制

在创建任务或拆解需求时，系统会自动评估任务点数：

```
拆解任务 T001
    │
    ├─► 预估点数 ──► 点数 > 5?
    │               ├─► 是 ──► 拆解为多个子任务
    │               │           ├─► 子任务1 (预估点数)
    │               │           ├─► 子任务2 (预估点数)
    │               │           └─► ...
    │               │
    │               └─► 否 ──► 停止拆解
    │
    └─► 递归检查每个子任务
          └─► 点数 > 5? ──► 继续拆解，直到点数 <= 5
```

**拆解规则**:
- 点数 > 5：将任务拆解为 2-5 个子任务
- 子任务递归进行拆解检查，直到所有叶子任务点数 <= 5
- 拆解时建立 parentId 和 childIds 关联
- 预估点数根据任务类型自动估算（见第 9 节）

## 3.2 任务领取机制

当用户命令开始执行但未指定具体任务时，自动选择任务：

```
领取任务
    │
    ├─► 扫描所有非 completed、非 abandoned 的任务
    │
    ├─► 寻找第一个 waiting_dep 或 pending 的任务
    │
    ├─► 递归查找最小任务
    │     │
    │     ├─► 该任务有依赖? ──► 顺着依赖链找到最底层未完成的任务
    │     │                     └─► 返回该依赖任务
    │     │
    │     └─► 该任务有子任务? ──► 顺着子任务链找到最底层未完成的任务
    │                             └─► 返回该子任务
    │
    ├─► 分配子代理 ──► 根据策略选择子代理
    │               └─► 记录 agentRef
    │
    └─► 领取该任务 (状态 → processing)
```

**优先级**:
1. waiting_dep 状态的任务优先
2. pending 状态的任务次之
3. 顺着依赖引用或子任务引用递归查找
4. 返回找到的最小任务（无未完成依赖、无未完成子任务）

**任务分配**:
- 领取任务时，自动分配给一个子代理
- 分配策略：轮询或根据子代理专业领域匹配
- 任务记录被分配的子代理编号 (agentRef)

## 3.3 新会话机制

使用 MCP 工具创建新会话，每个子任务使用子代理完成：

```
执行任务前
    │
    └─► MCP 工具调用 ──► 创建新会话
                            │
                            ├─► 继承项目上下文
                            ├─► 继承任务详情
                            ├─► 启动子代理执行任务
                            └─► 任务完成后返回主会话
```

**触发时机**:
- 阶段切换时（需求分析 → 计划制定 → 任务执行）
- 每个子任务执行前
- 每次重试时

**目的**:
- 避免上下文污染
- 每个任务有独立的执行环境
- 便于追踪和调试
- 会话结束后自动返回主会话

## 4. 自愈循环机制

```
任务失败
    │
    ▼
┌───────────────────────┐
│ 1. 记录失败日志        │
│ 2. 根因分析 (LLM)     │
│ 3. 生成修复方案       │
│ 4. 执行修复           │
└───────────────────────┘
    │
    ▼
┌───────────────────────┐ 是否需要
│ 检查:用户介入 │
│ (权限问题、人工授权等) │
└───────────────────────┘
    │ 是
    ▼
┌───────────────────────┐
│ 询问用户              │
├─► 用户授权/回答 ──► 继续执行
├─► 用户拒绝 ──► 停止，标记 failed
└───────────────────────┘
    │ 否
    ▼
┌───────────────────────┐
│ 检查: 是否达到最大重试 │──是──► 标记为 failed，记录原因
│        或无法继续？    │
└───────────────────────┘
    │ 否
    ▼
┌───────────────────────┐
│ 重试 (回到执行步骤)    │
└───────────────────────┘
```

**退出条件**:
- 任务完成
- 达到最大重试次数
- 用户拒绝继续

**失败日志存储**:
- 位置: `.zerocode/logs/T001/`
- 包含: 错误信息、根因分析、修复尝试记录

### 5.5 长期记忆系统

每个子代理拥有独立的长期记忆系统，用于存储执行任务过程中积累的经验和知识。

#### 5.5.1 记忆存储结构

```
.zerocode/agents/
├── A001/
│   ├── profile.md          # 代理基本资料
│   ├── memory/            # 长期记忆
│   │   ├── experiences/    # 经验记录
│   │   │   └── 2026-03-05-experience-001.md
│   │   ├── lessons/        # 教训总结
│   │   │   └── 2026-03-05-lesson-001.md
│   │   └── solutions/      # 问题解决方案
│   │       └── 2026-03-05-solution-001.md
│   └── reflection/        # 复盘记录
```

#### 5.5.2 记忆类型

| 类型 | 描述 | 示例 |
|------|------|------|
| experiences | 执行任务的经验 | "在实现登录功能时，使用 JWT 比 Session 更好" |
| lessons | 教训总结 | "不要在生产环境直接修改数据库表结构" |
| solutions | 问题解决方案 | "当 TypeScript 编译报错时，先检查 tsconfig.json 配置" |

#### 5.5.3 记忆文件格式

```markdown
---
id: EXP-001
type: experience
tags:
  - 认证
  - JWT
  - 安全
createdAt: 2026-03-05T10:00:00Z
taskRef: T001
---

# 经验：使用 JWT 进行用户认证

## 问题背景
用户登录功能需要支持移动端和 Web 端...

## 解决方案
1. 使用 JWT Token 进行无状态认证
2. Token 有效期设置为 24 小时
3. 使用 Refresh Token 续期

## 效果
- 实现了跨平台支持
- 减少了服务器存储压力
```

#### 5.5.4 记忆检索

- **语义搜索**: 使用关键词或自然语言描述检索相关记忆
- **标签筛选**: 按记忆类型、标签、时间筛选
- **相关任务**: 检索与当前任务相关的历史经验

    │
    ▼

    ▼

    │
    ▼

## 9. 点数估算

根据任务描述让 AI 估算点数，并参考历史任务的实际消耗：

- **AI 估算** - 基于任务描述让 AI 估算点数
- **历史参考** - 根据类似历史任务的实际 Token 消耗调整
- **点数用途** - 估算任务所需的 Token 消耗

**估算方法**:
- 代码类任务：根据预估代码量估算
- 文档类任务：根据文档长度估算
- 配置类任务：固定点数
- 可配置调整系数

## 10. 设计/需求/任务修改感知机制

通过 Claude Code Hooks 实现，当用户修改了项目的需求文档、设计文档或任务文档时，自动触发以下流程：

### 10.1 触发条件

Hook 监听以下文件变化：
- `.zerocode/requirements/*.md` - 需求文档
- `docs/plans/*.md` - 计划/设计文档
- 项目根目录下的设计文档（如 `docs/design/*.md`、`PROJECT.md`）

**注意**：不感知任务文档（`.zerocode/tasks/*.md`）的修改

**触发时机**: 用户保存文件时立即触发
**通知方式**: 在当前会话中通知用户

### 10.2 处理流程

```
检测到文件变化
    │
    ├─► 分析改动内容 ──► 提取改动摘要
    │
    ├─► 判断是否需要头脑风暴 ──► 改动对现有设计有影响?
    │                                   │
    │                                   ├─► 是 ──► 触发头脑风暴
    │                                   │           └─► 向用户提问，补充遗漏信息
    │                                   │
    │                                   └─► 否 ──► 继续
    │
    └─► 找出关联任务 ──► 遍历所有任务，找出引用该文档的任务
              │
              └─► 判断任务状态
                  │
                  ├─► processing / completed / abandoned
                  │     └─► 创建新任务满足改动需求
                  │
                  └─► pending / waiting_dep
                        │
                        └─► 判断改动大小
                            │
                            ├─► 小改动（命名/数值/配置值）
                            │     └─► 直接修改任务相关内容
                            │
                            └─► 大改动（难以兼容）
                                  └─► 废弃原有相关任务
                                      └─► 重新创建新任务

```

### 10.3 改动大小判断

| 改动类型 | 分类 | 处理方式 |
|----------|------|----------|
| 变量名、函数名修改 | 小改动 | 直接修改任务内容 |
| 配置值、数值调整 | 小改动 | 直接修改任务内容 |
| 新增功能点 | 大改动 | 废弃旧任务，重新创建 |
| 删除功能点 | 大改动 | 废弃旧任务，重新创建 |
| 逻辑流程变更 | 大改动 | 废弃旧任务，重新创建 |
| 接口/数据结构变更 | 大改动 | 废弃旧任务，重新创建 |

### 10.4 Skill 新增

| Skill | 功能 |
|-------|------|
| `zerocode:hook-enable` | 启用文件变更监听 Hook |
| `zerocode:hook-disable` | 禁用文件变更监听 Hook |
| `zerocode:diff-analyze` | 手动分析某个改动的关联任务 |

## 11. 初始化

### 11.1 初始化时机

- **手动触发**: 用户执行 `/zerocode:init` 命令
- **首次执行时自动触发**: 第一次执行任务时自动初始化

### 11.2 初始化内容

创建 `.zerocode/` 目录结构：
```
.zerocode/
├── tasks/          # 任务文档
├── requirements/   # 需求文档
├── agents/         # 子代理文档
│   └── index.json  # 子代理索引
├── design.md       # 设计文档
└── logs/           # 执行日志
```

### 11.3 预制子代理

根据项目类型可配置生成不同子代理：
- 默认: 产品经理、技术经理、程序员
- 可自定义子代理角色和专业领域
